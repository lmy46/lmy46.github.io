Spring 核心 02 - Bean 的定義與註冊

發表於2025-09-07|更新於2025-09-29|JavaSpringSpring 核心實戰
|字數總計:1.6k|閱讀時長:5分鐘|閱讀量:

前言：IoC 容器的「員工名冊」

在上一篇文章中，我們將 ApplicationContext (IoC 容器) 比喻為應用程式的「總經理」。這位總經理神通廣大，能自動化地管理我們應用程式中所有物件（員工）的生命週期與協作關係。

但一個根本的問題是：這位總經理，是如何拿到最初的「員工名冊」的？它怎麼知道要去「聘用」UserService 和 UserRepository，而不是其他普通的 Java 物件？

Spring 提供了兩種主要的方式來建立這份名冊，也就是定義 Bean：

  1. **元件掃描 (Component Scanning)**：類似於員工的「自主報名」。我們在類別上貼一個標籤（註解），IoC 容器在啟動時會自動掃描並將其納入管理。
  2. Java 組態 (Java Configuration)：類似於一份「集中招聘清單」。我們在一個專門的組態檔中，明確地告訴 IoC 容器要去聘用誰，以及該如何聘用（實例化）他們。

本篇文章將帶您深入探討這兩種核心的 Bean 定義方式，並分析它們各自的優劣與最佳適用場景。

方法一：元件掃描 (Component Scanning)

這是我們在「Spring Series 基礎」中一直以來所使用的方式，也是 Spring Boot 應用中最常見的方式。

運作原理：
這個機制的起點是主應用程式類別上的 @SpringBootApplication 註解。我們在第 2 篇中學過，這個註解內部包含了 @ComponentScan。@ComponentScan 會指示 Spring IoC 容器，去掃描主應用程式所在的套件（以及所有子套件），找出所有被「刻板印象註解」標示的類別，並自動將它們註冊為 Bean。

這些「刻板印象註解」包括：

  • @Component：最通用的註解，表示這是一個 Spring 元件。
  • @Service：用於標示業務邏輯層。
  • @Repository：用於標示資料存取層。
  • @RestController / @Controller：用於標示 Web 控制器層。
  • @Configuration：用於標示這是一個組態類別（我們稍後會詳談）。

優點：

  • 方便快捷：只需在類別上加一個註解即可，極大地減少了組態的複雜性。
  • 去中心化：Bean 的定義與其類別實作放在一起，易於理解。

缺點：

  • 侵入性：我們必須能夠修改原始碼，才能為類別加上註解。這意味著我們無法用這種方式，來將第三方函式庫中的類別註冊為 Bean。

方法二：Java 組態 (Java Configuration)

當我們需要將第三方函式庫的類別（例如 ObjectMapper），或者需要進行複雜初始化操作的物件交給 Spring 管理時，Java 組態就派上用場了。

運作原理：

  1. 我們建立一個被 @Configuration 註解標示的類別。
  2. 在這個類別中，我們建立一個或多個被 @Bean 註解標示的方法。
  3. Spring IoC 容器會自動呼叫這些 @Bean 方法，並將方法的回傳值，作為一個 Bean 註冊到容器中。

  • 程式碼詳解：@Bean 註解告訴 Spring：「請執行 objectMapper() 這個方法，並將它回傳的那個 ObjectMapper 物件，當作一個 Bean 來管理」。方法的名稱 objectMapper 預設會成為這個 Bean 的 ID。

優點：

  • 完全的控制權：您可以在 @Bean 方法中執行任意的 Java 程式碼，來建立和設定您的物件。
  • 非侵入性：可以為任何類別（包括您無法修改原始碼的第三方類別）建立 Bean。
  • 集中管理：相關的 Bean 定義可以集中在同一個組態類別中，一目了然。

如何選擇？最佳實踐

在真實專案中，這兩種方式是並存且互補的。

  • **使用元件掃描 (@Component, @Service 等)**：

    • 用於您自己專案中的業務元件（Controllers, Services, Repositories 等）。這是主要的、最常見的方式。

  • **使用 Java 組態 (@Configuration, @Bean)**：

    • 用於將第三方函式庫中的類別實例化為 Bean。
    • 用於當 Bean 的建立過程非常複雜，需要執行一段初始化邏輯時。

使用 @Import 實現組態模組化

總結

在本篇文章中，我們深入探討了 Spring IoC 容器建立「員工名冊」的兩種核心方式。

  • 元件掃描：透過 @Component 及其衍生註解，實現 Bean 的「自主報名」，方便快捷，適用於我們自己的業務元件。
  • Java 組態：透過 @Configuration 與 @Bean，實現 Bean 的「集中定義」，擁有完全的控制權，適用於第三方元件或複雜的建立邏輯。

我們現在已經完全理解了 Spring IoC 容器是如何「找到」並「定義」它要管理的 Beans。但在容器內部，這些 Beans 還有著不同的「生命型態」。在下一篇文章中，我們將深入探討 **Bean 的作用域 (Scope)**，學習為何有些 Bean 是全域唯一的，而有些卻可以每次請求都產生一個新的。
