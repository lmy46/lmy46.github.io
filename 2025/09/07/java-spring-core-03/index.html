Spring 核心 03 - 精通 Bean 的作用域 (Scope)

發表於2025-09-07|更新於2025-09-29|JavaSpringSpring 核心實戰
|字數總計:2.6k|閱讀時長:8分鐘|閱讀量:

前言：Bean 的「生命型態」

我們在先前的範例中，將 UserService 注入到 UserController 中。如果我們再建立一個 OrderController，並同樣注入 UserService，那麼這兩個 Controller 得到的，是同一個 UserService 物件實例，還是兩個不同的新物件呢？

答案是：預設情況下，它們得到的是同一個、全域唯一的物件。

這種物件在容器中的「生命週期」與「可見範圍」，就是由 Bean 的作用域 (Scope) 來決定的。將 IoC 容器比喻成一個工具棚，那麼：

  • singleton 作用域的 Bean，就像掛在牆上的一把共用鐵鎚。任何人需要時，都來拿這同一把鐵鎚去用。
  • prototype 作用域的 Bean，則像工具棚裡的一盒一次性手套。每個人來拿的時候，都是從盒子裡取一雙全新的手套。

理解不同的作用域，並為您的 Bean 選擇正確的「生命型態」，是撰寫高效、安全且可預測程式碼的關鍵。

singleton - 全域唯一的共享實例

singleton 是 Spring Bean 的預設作用域。這意味著對於任何一個 Bean 的定義，Spring IoC 容器只會建立一個實例。無論這個 Bean 在應用程式的哪個角落被注入或請求，容器回傳的，永遠都是指向同一個共享物件的參考。

為何 singleton 是預設？
因為它非常高效。在我們的應用程式中，絕大多數的 Bean，如 Services, Repositories, Configurations 等，都是**無狀態的 (Stateless)**。它們本身不儲存與特定使用者或特定請求相關的資料，只提供可重複使用的方法。因此，讓一個單一的實例，被整個應用程式共享並同時服務於成千上萬的請求，是既安全又節省資源的做法。

我們可以透過一個簡單的測試來證明。

prototype - 每次請求都建立新實例

與 singleton 相反，當您將一個 Bean 的作用域定義為 prototype 時，IoC 容器會在每一次該 Bean 被請求（注入或透過 getBean() 呼叫）時，都建立一個全新的實例。

如何定義：

與 singleton 的關鍵差異：生命週期管理
對於 singleton Bean，Spring 容器會管理其完整的生命週期（從建立到銷毀）。但對於 prototype Bean，Spring 在建立、設定並初始化完畢，將其交給客戶端程式碼後，就不再追蹤這個物件了。這意味著 Spring 不會為 prototype Bean 呼叫銷毀相關的回呼方法（如 @PreDestroy）。

適用場景：
prototype 主要用於那些有狀態的 (Stateful) Bean。例如，一個物件需要為某個特定的、非 Web 的任務儲存中間狀態，並且不希望這個狀態被其他任務共享。

Web 應用中的作用域

在 Web 應用程式中，Spring 提供了幾個與 HTTP 請求生命週期相關的額外作用域：

  • request：為每一次 HTTP 請求，建立一個新的 Bean 實例。該實例僅在當前請求內有效，請求結束後即被銷毀。

    • 適用場景：儲存與單次請求相關的資料，例如使用者的 IP 位址、請求的追蹤 ID 等。

  • session：為每一個 HTTP Session，建立一個新的 Bean 實例。在同一個使用者的 Session 期間，多次請求都會共用這一個實例。

    • 適用場景：最經典的例子就是購物車。每個使用者的購物車物件都應該是獨立的，並且在他們瀏覽網站的整個會話期間都保持存在。

  • **application**：為整個 Web 應用程式（ServletContext 的生命週期）建立一個單一的 Bean 實例。行為類似 singleton，但與 Web 上下文綁定。

核心難題：長短生命週期的依賴注入

現在，問題來了：我們公司（Spring 應用）在成立之初，就需要任命一位 singleton 的「專案經理」（例如我們的 UserController）。公司章程規定，任命經理時，必須立刻指派一位「任務承辦人」（例如一個 request 作用域的 RequestDataHolder Bean）給他。

這顯然是不可能的！因為公司剛成立（應用程式剛啟動）時，根本還沒有任何客戶請求（HTTP Request）進來，自然也就不存在任何「任務承辦人」。如果您嘗試這樣做，應用程式在啟動時就會因無法解析依賴而報錯。

解決方案：作用域代理 (Scoped Proxy) 的智慧

為了解決這個矛盾，Spring 提供了一個極其聰明的設計：**作用域代理 (Scoped Proxy)**。

這個方案可以比喻為：我們不直接給「專案經理」一位固定的「任務承辦人」，取而代之，我們給他一位**「萬能秘書」(Proxy 代理)**。這位秘書本身也是 singleton 的，可以在公司成立時就跟著經理一起到職。他的職責很簡單：當經理需要承辦人時，這位秘書會即時地去找到「當下」正在處理這個請求的那位承辦人，然後把工作交辦給他。

實戰演練

  1. 任務承辦人 (RequestDataHolder)
    我們建立一個 request 作用域的 Bean，並透過 proxyMode 告訴 Spring：「別直接給我這個物件，請給我一位『秘書』」。

  2. 專案經理 (ScopeController)
    這個 Controller 是 singleton 的，它注入的 requestDataHolder 實際上是那個「萬能秘書」代理。

運作流程：

  • 應用程式啟動時：ScopeController 被建立。Spring 發現它需要 RequestDataHolder，並且看到 proxyMode 設定，於是動態產生一個 RequestDataHolder 的代理類別，並將這個代理物件注入到 Controller 中。啟動成功。
  • 當請求 A 進入時：Spring 為請求 A 建立一個真實的 RequestDataHolder 實例 A。當 testScope 方法被執行，呼叫 requestDataHolder.setData() 時，注入的代理物件會被觸發，它動態地找到屬於請求 A 的實例 A，並將方法呼叫委派給它。
  • 當請求 B 進入時：Spring 為請求 B 建立另一個真實的 RequestDataHolder 實例 B。此時，ScopeController 中同一個代理物件，會聰明地將方法呼叫委派給屬於請求 B 的實例 B。

企業級實戰場景

這種「請求作用域 + 代理」的模式，在企業級應用中極為普遍，主要用於處理**「與單次請求相關的、需要在不同層級間共享的狀態」**。

  • **存放當前登入使用者資訊 (CurrentUserHolder)**：在請求最開始的 Filter/Interceptor 中解析 Token 並將使用者資訊存入，之後在任何 Service 或 Repository 層都可以安全地注入並獲取當前操作者，用於權限判斷或日誌記錄。
  • **請求級別的追蹤 ID (RequestTrace)**：在微服務中，為每個請求生成一個 traceId 並存入，然後在 Logback 的 MDC 中取出，實現全鏈路日誌追蹤。

設計原則：為何選擇最短的生命週期？

為什麼我們不乾脆用 session 作用域來儲存這些資訊呢？選擇滿足需求的最短生命週期，是基於以下關鍵原因：

特性 request session singleton
生命週期 極短 (單次 HTTP 請求) 中等 (使用者會話期間) 極長 (整個應用程式)
狀態儲存 單次請求相關的狀態 儲存單一使用者跨請求的狀態 理想情況下應為無狀態
記憶體佔用 極小，立即釋放 持續佔用，使用者越多壓力越大 僅一個實例，記憶體固定
執行緒安全 天生安全 (每個執行緒獨享) 非安全 (同 Session 多請求共享) 非安全 (所有執行緒共享)
資料一致性 最高 (每次都是最新資料) 可能存在過期資料 -

總結來說：request 作用域以其狀態隔離、執行緒安全和記憶體效率的絕對優勢，成為處理「一次性」請求資料的最佳實踐。

總結

在本篇文章中，我們深入了 Bean 作用域的內部世界。

  • 我們不僅理解了 singleton, prototype, request, session 的行為差異。
  • 更重要的是，我們透過「專案經理與萬能秘書」的比喻與實戰程式碼，徹底掌握了作用域代理如何解決長短生命週期 Bean 的依賴注入難題。
  • 我們還探討了 request 作用域在企業級應用中的實際用途與其背後的設計原則。

我們已經了解了 Bean 的定義方式與不同的生命型態 (Scope)。但一個 Bean 從被 Spring 容器實例化，到最終被銷毀，中間還經歷了一系列複雜的過程。在下一篇文章中，我們將深入探討 Bean 的完整生命週期，學習如何使用 @PostConstruct 等回呼機制，在 Bean 生命的關鍵時刻執行我們的自訂邏輯。
