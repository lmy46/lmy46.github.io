交易管理 05 - 隔離級別 (Isolation) 與併發問題

發表於2025-09-09|更新於2025-10-25|JavaSpringSpring 交易管理實戰
|字數總計:1.5k|閱讀時長:5分鐘|閱讀量:

前言：當多個使用者同時操作

想像一個共享的線上文件，例如 Google Docs。當您正在編輯某個段落時，您肯定不希望其他協作者看到您打到一半、語意不通的句子。同樣地，當一個高併發的電商系統在處理成千上萬的訂單時，多個交易可能會同時嘗試讀取和修改同一個商品的庫存。

如果沒有任何保護措施，這種併發操作將會導致災難性的資料錯亂。**交易的隔離性 (Isolation)**，也就是 ACID 原則中的「I」，正是為了解決這個問題而存在的。它定義了一個交易在執行時，能在多大程度上「看見」其他併發交易所做的、尚未提交的修改。

本篇文章將帶您深入理解在高併發下，可能出現的三大「併發讀取問題」，並學習 SQL 標準所定義的四種隔離級別是如何逐一解決這些問題，以及如何在 Spring 中設定它們。

併發下的三大讀取問題

1. 髒讀 (Dirty Read)

  • 概念：一個交易（T1）讀取到了另一個併發交易（T2）已經修改但尚未提交的資料。如果 T2 最終選擇了回滾，那麼 T1 讀取到的就是一筆從未真實存在過的「髒」資料。
  • 場景：
    時間 交易 A (更新使用者名稱) 交易 B (讀取使用者名稱)
    1 BEGIN
    2 UPDATE users SET name = '新名字'
    WHERE id = 1
    3 BEGIN
    4 SELECT name FROM users
    WHERE id = 1 (讀到 ‘新名字’)
    5 ROLLBACK (操作取消)
    6 COMMIT
  • 後果：交易 B 帶著 ‘新名字’ 這個錯誤的資料繼續執行下去，造成了資料不一致。

2. 不可重複讀 (Non-Repeatable Read)

  • 概念：在同一個交易中，兩次執行完全相同的查詢，但第二次查詢的結果與第一次不同。這是因為在這兩次查詢之間，有另一個交易更新了這筆資料並已提交。
  • 重點：問題的核心在於**修改 (UPDATE)**。
  • 場景：
    時間 交易 A (查詢商品價格) 交易 B (調整商品價格)
    1 BEGIN
    2 SELECT price FROM products
    WHERE id = 1 (得到 100 元)
    3 BEGIN
    4 UPDATE products SET price = 120
    WHERE id = 1
    5 COMMIT
    6 SELECT price FROM products
    WHERE id = 1(再次查詢，得到 120 元)
    7 COMMIT
  • 後果：交易 A 在同一個交易內，對同一個商品的價格得到了兩個不同的結果，導致其後續的計算或判斷可能出現邏輯錯誤。

3. 幻讀 (Phantom Read)

  • 概念：在同一個交易中，兩次執行完全相同的範圍查詢，但第二次查詢回傳的紀錄筆數與第一次不同。這是因為在這兩次查詢之間，有另一個交易新增 (INSERT) 或刪除 (DELETE) 了符合該範圍條件的資料並已提交。
  • 重點：問題的核心在於新增或刪除，導致紀錄「像幻影一樣」憑空出現或消失。
  • 場景：
    時間 交易 A (統計 VIP 使用者數量) 交易 B (新增 VIP 使用者)
    1 BEGIN
    2 SELECT COUNT(*) FROM users
    WHERE level = 'VIP' (得到 10 人)
    3 BEGIN
    4 INSERT INTO users (..., level)
    VALUES (..., 'VIP')
    5 COMMIT
    6 SELECT COUNT(*) FROM users
    WHERE level = 'VIP' (再次統計，得到 11 人)
    7 COMMIT
  • 後果：交易 A 的統計結果出現不一致，如果後續業務依賴於這個統計數字，將會產生錯誤。

四種隔離級別的權衡

為了解決上述問題，SQL 標準定義了四種隔離級別。每個級別提供了不同程度的保護，但同時也帶來了不同程度的效能開銷。隔離級別越高，資料一致性越強，但系統的併發處理能力就越低。

隔離級別 解決了髒讀？ 解決了不可重複讀？ 解決了幻讀？ 效能/併發性
READ_UNCOMMITTED ❌ ❌ ❌ 最高
READ_COMMITTED ✅ ❌ ❌ 較高
REPEATABLE_READ ✅ ✅ ❌ 較低
SERIALIZABLE ✅ ✅ ✅ 最低
  • **READ_COMMITTED (讀已提交)**：大多數資料庫（如 PostgreSQL, Oracle, SQL Server）的預設級別。它提供了防止髒讀的基本保障，在一致性與效能之間取得了良好的平衡。
  • **REPEATABLE_READ (可重複讀)**：MySQL InnoDB 引擎的預設級別。它能解決不可重複讀的問題，提供了更高的一致性保障。

如何在 Spring 中設定隔離級別

在 Spring 中設定隔離級別非常簡單，只需在 @Transactional 註解中加入 isolation 屬性即可。

假設我們有一個產生財務報表的方法，這個方法需要多次查詢資料庫，並且絕對不希望在查詢過程中，資料被其他交易修改。此時，REPEATABLE_READ 就是一個很好的選擇。

注意：您在 Spring 中設定的隔離級別，只是一個「請求」。底層的資料庫必須支援該隔離級別，設定才會生效。
