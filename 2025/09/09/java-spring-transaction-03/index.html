交易管理 03 - 常見失效場景分析

發表於2025-09-09|更新於2025-10-25|JavaSpringSpring 交易管理實戰
|字數總計:1.5k|閱讀時長:5分鐘|閱讀量:

前言：我的 @Transactional 為何沒有作用？

這幾乎是每一位 Spring 開發者都曾遇到的靈魂拷問：「我明明在方法上加了 @Transactional，也確實拋出了例外，為什麼資料庫裡的資料沒有回滾？」

這個問題的答案，99% 的情況下都與我們在上一篇學到的核心原理有關——AOP 動態代理。Spring 的交易「魔法」只在方法的呼叫經過代理物件時才會生效。如果您的呼叫方式不小心「繞過」了代理，那麼交易自然也就不會開啟。

本篇文章將像一份「偵錯手冊」，為您詳細剖析三種最經典的 @Transactional 失效場景，並從 AOP 代理的視角，解釋其根本原因與提供解決方案。掌握了這些，您將能自信地診斷與修復絕大多數的交易問題。

失效場景一：同類別內部方法呼叫 (自調用問題)

這是最常見、也最隱晦的失效場景。

問題描述：在一個 Service 類別中，一個沒有交易註解的方法 A()，去呼叫了同一個類別中、被 @Transactional 標註的方法 B()。結果發現，方法 B() 的交易沒有生效。

程式碼範例：

根本原因：呼叫繞過了代理
回想一下，Controller 注入的 UserService 實際上是代理物件。

  • 外部呼叫：當 Controller 呼叫 userService.methodA() 時，請求會先經過代理物件。代理物件發現 methodA() 沒有 @Transactional 註解，於是它不會開啟交易，直接呼叫真實 UserService 物件的 methodA()。
  • 內部呼叫：接著，在真實物件的 methodA() 內部，程式執行到 this.methodB()。這裡的 this 關鍵字，指向的是真實的 UserService 物件本身，而不是代理物件。因此，這個呼叫是一個類別內部的普通方法呼叫，它完全沒有機會再經過外部的代理物件，代理也就無法為 methodB() 加上交易邏輯。

解決方案：
核心思想是：必須讓對 methodB() 的呼叫，也經過代理物件。

  1. 最佳實踐：拆分到不同的 Bean
    這是最清晰、最符合單一職責原則的做法。將 methodB() 拆分到一個新的 ServiceB 中，然後在 ServiceA 中注入 ServiceB 來呼叫它。

  2. 權宜之計：注入自身代理 (Self-Injection)
    在某些情況下，可以讓 Bean 自己注入自己的代理。

失效場景二：非 public 方法

問題描述：@Transactional 註解被加在了 private 或 protected 方法上。

程式碼範例：

根本原因：Spring AOP 預設使用的 CGLIB 動態代理，是透過繼承目標類別並覆寫 (override) 其 public 方法來實現的。Java 的繼承規則決定了 private 方法無法被子類別覆寫，因此代理物件根本「看不見」也無法攔截到 private 方法的呼叫。

解決方案：非常簡單，永遠只將 @Transactional 註解用在 public 方法上。

失效場景三：錯誤的例外處理

根本原因：Spring 的交易代理，預設只會在捕捉到未受檢例外 (Unchecked Exception)，也就是 RuntimeException 及其子類別，或是 Error 時，才會觸發交易回滾。對於受檢例外 (Checked Exception)，例如 IOException, SQLException，它預設不會觸發回滾。

總結

在本篇文章中，我們深入分析了 @Transactional 最常見的三大失效場景。

  1. 自調用問題：同類別內部呼叫會繞過代理，應透過拆分 Bean 或注入自身代理來解決。
  2. 方法可見性：@Transactional 必須用在 public 方法上。
  3. 例外處理：確保會觸發回滾的例外，能夠成功地從交易方法中拋出，並為受檢例外配置 rollbackFor 規則。

請牢記核心原理：交易是由外部的代理物件所控制的。只要確保您的方法呼叫會經過這個代理，您的交易就能如預期般運作。
