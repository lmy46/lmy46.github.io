交易管理 06 - 進階主題與最佳實踐

前言：從精通到卓越

恭喜您來到「交易管理實戰」的最後一站！我們已經一起探索了 AOP 代理的奧秘，穿越了傳播行為與隔離級別的複雜迷宮，也學會了如何避開常見的失效陷阱。現在的您，已經對 Spring 的交易管理有了深刻的理解。

在本篇最終章中，我們將為您的技能庫，再添增幾件強大的「神兵利器」。我們將學習如何透過 readOnly 屬性榨取查詢效能、如何使用 timeout 為交易加上保險絲，並探索在特殊場景下，比註解更靈活的 TransactionTemplate 程式化交易，以及實現終極解耦的 @TransactionalEventListener 事件驅動模式。

readOnly 屬性 - 不只是個標記

我們在基礎系列中提過，查詢方法應該加上 @Transactional(readOnly = true)。現在，讓我們深入其背後的原因。這不僅僅是一個語義上的標記，它能帶來實質的效能優化。

優化原理：

  1. JPA/Hibernate 層面：當您將交易標示為 readOnly，等於是告訴了底層的 Hibernate：「在這個交易裡，我保證不會有任何資料修改」。Hibernate 收到這個訊號後，就可以省略掉一個非常耗費效能的步驟——**髒檢查 (Dirty Checking)**。在一個讀寫交易結束時，Hibernate 需要檢查所有被載入到記憶體中的 Entity 物件，看它們的狀態是否被修改過，以便將變更同步回資料庫。在唯讀交易中，這個檢查就可以被完全跳過。
  2. JDBC/資料庫層面：這個 readOnly 標記會一路傳遞到 JDBC 驅動程式，甚至資料庫本身。某些資料庫（尤其是在讀寫分離的架構中）可以利用這個提示，將查詢路由到負載較輕的唯讀副本 (Read Replica) 資料庫上，從而減輕主資料庫的壓力。

最佳實踐：任何一個只包含查詢操作、不涉及任何資料修改的 Service 方法，都應該毫不猶豫地加上 @Transactional(readOnly = true)。

timeout 屬性 - 為您的交易加上保險絲

問題：如果一個交易因為某些原因（例如，等待外部服務回應、資料庫死結）而被長時間卡住，它將會長時間地佔用資料庫連線與鎖定資源，最終可能拖垮整個應用程式。

解決方案：為交易設定一個超時時間。

timeout 屬性就像一個保險絲，它為您的交易提供了一個執行時間的上限，是防止單一失控交易影響整個系統穩定性的重要保障。

程式化交易管理：TransactionTemplate

@Transactional 註解非常方便，但它的作用粒度是整個方法。如果我們只想在一個長方法的中間一小段程式碼中啟用交易，或者交易的邊界需要由複雜的動態邏輯來決定時，該怎麼辦？

答案是使用程式化交易管理——TransactionTemplate。

終極解耦：@TransactionalEventListener

比喻：訂單處理流程

想像一下，你是一家線上商店的老闆，正在處理一筆新的顧客訂單。你的訂單處理流程有三個主要步驟：

  1. 建立訂單：在資料庫中新增一筆訂單記錄。
  2. 發送通知：發送簡訊或電子郵件給顧客，確認訂單已成立。
  3. 記錄日誌：在日誌系統中記錄這筆訂單的詳細資訊，供後續分析。

  • 這意味著，雖然資料庫中的訂單記錄被取消了，但 顧客可能已經收到了「訂單已成立」的通知簡訊。這會造成資料不一致，讓顧客感到困惑。

使用 @TransactionalEventListener 的解決方案

@TransactionalEventListener 就像是訂單處理流程中的一個「守門員」。它會告訴程式：「等一下！先不要寄簡訊和寫日誌。等我確認這筆訂單在資料庫中真的、確實地建立了，我們再做後續的事情。」

這個「守門員」的機制就是 TransactionPhase.AFTER_COMMIT。它確保被標註的方法只會在 交易成功提交後 才執行。
