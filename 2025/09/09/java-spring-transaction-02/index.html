交易管理 02 - 核心原理：ACID 與 AOP 代理

發表於2025-09-09|更新於2025-10-25|JavaSpringSpring 交易管理實戰
|字數總計:1.6k|閱讀時長:4分鐘|閱讀量:

前言：魔法的背後

在「Spring Series 基礎」中，我們學會了在 Service 方法上加上一個簡單的 @Transactional 註解，就能神奇地讓一系列資料庫操作具備「要麼全成功，要麼全失敗」的特性。

但是，您是否曾想過這背後的原理？

  • 一個 Java 註解，是如何去控制遠端資料庫的 BEGIN, COMMIT, ROLLBACK 命令的？
  • Spring 究竟是在什麼時機開啟交易，又是在什麼時機提交或回滾的？

理解這些問題的答案，是從「會用」@Transactional 邁向「精通」它的第一步。本篇文章將帶您深入兩個核心主題：首先，我們將鞏固對交易目標 (ACID 原則) 的理解；接著，我們將揭密 Spring 實現這一目標的**機制 (AOP 動態代理)**。

交易的基石：深入 ACID 原則

ACID 是任何一個可靠的交易所必須遵循的四大特性，它們是資料庫對我們許下的「承諾」。

1. 原子性 (Atomicity)

  • 概念：「All or Nothing」，不可分割。
  • 比喻：銀行轉帳操作包含「從 A 帳戶扣款」和「向 B 帳戶存款」兩個步驟。原子性確保了這兩個步驟要麼都成功，要麼如果中間發生任何錯誤（如系統斷電），已經完成的「扣款」操作也必須被撤銷（回滾），使得帳戶回到交易前的狀態。
  • 對應 Spring：@Transactional 所包裹的整個方法，就是一個原子操作單元。

2. 一致性 (Consistency)

  • 概念：交易的執行，必須使資料庫從一個有效的、符合所有業務規則的狀態，轉變到另一個有效的狀態。
  • 比喻：在轉帳前，A 和 B 帳戶總金額為 50000 元。無論交易如何進行，交易結束後，兩者的總金額依然必須是 50000 元。資料庫的完整性約束（如 NOT NULL）也必須始終被滿足。
  • 對應 Spring：如果您的業務邏輯會導致資料狀態不一致（例如，建立一個 username 為 null 的 User 實體，而資料庫欄位不允許為空），交易在提交時將會失敗並回滾。

3. 隔離性 (Isolation)

  • 概念：當多個交易併發執行時，它們的執行過程應該是相互隔離的，感覺不到彼此的存在。
  • 比喻：在 A 向 B 轉帳的過程中（交易尚未提交），另一個交易（例如，計算銀行總存款）不應該讀取到這個「中間狀態」（A 的錢少了，B 的錢還沒多）。
  • 對應 Spring：這是交易管理中最複雜的部分，@Transactional 的 isolation 屬性專門用來控制隔離的級別。我們將在後續的 交易管理 05 篇中深入探討。

4. 持久性 (Durability)

  • 概念：一旦交易成功提交，它對資料庫的修改就是永久性的。
  • 比喻：銀行系統通知您「轉帳成功」的那一刻，這個結果就是板上釘釘了。即使下一秒整個機房斷電，重啟後這筆交易紀錄依然會存在。
  • 對應 Spring：當 @Transactional 方法成功執行完畢並提交後，資料將被安全地寫入到資料庫的磁碟（或其他持久化儲存）中。

揭密 Spring 的魔法：AOP 動態代理

理解了 ACID 這個「目標」，我們現在來揭示 Spring 實現這個目標的「機制」。答案，就在我們之前學過的 **AOP (切面導向程式設計)**。

當 Spring 容器在啟動時，偵測到您的某個 Bean（例如 UserService）中，有被 @Transactional 註解標示的 public 方法時，它並不會直接將您的 UserService 原始物件放入容器。取而代之的，是執行以下操作：

  1. 建立代理物件：Spring 在背後，使用 AOP 技術，動態地建立一個 UserService 的代理 (Proxy) 物件。這個代理物件擁有與 UserService 完全相同的方法，並且包裹住了真實的 UserService 物件。

  2. 注入代理物件：當 UserController 透過 @Autowired 請求注入 UserService 時，Spring 容器給它的，是那個代理物件，而不是真實的物件。

  3. 攔截方法呼叫：當 UserController 呼叫 userService.createUser() 時，這個呼叫首先會觸及代理物件。

  4. 執行交易邏輯：代理物件的 createUser() 方法，就像一個精密的保全系統，它會執行以下流程：

    • **(方法執行前)**：檢查 @Transactional 的設定，然後連接資料庫，開啟一個新的交易 (BEGIN TRANSACTION)。
    • (委派執行)：接著，代理物件才會去呼叫被它包裹的真實 UserService 物件的 createUser() 方法，執行我們撰寫的核心業務邏輯。
    • **(方法執行後)**：
      • 如果真實方法順利執行完畢，沒有拋出任何（會觸發回滾的）例外，代理物件就會向資料庫發出 COMMIT 命令，提交交易。
      • 如果真實方法在執行過程中拋出了 RuntimeException 或 Error，代理物件會捕捉到這個例外，然後向資料庫發出 ROLLBACK 命令，撤銷在此交易中的所有操作。

流程圖：
外部呼叫 -> [UserService 代理] -> 開啟交易 -> [真實 UserService] -> 執行業務邏輯 -> [真實 UserService] -> [UserService 代理] -> 提交/回滾交易 -> 返回結果/拋出例外

這就是 @Transactional 的秘密！所有交易管理的邏輯，都發生在您看不見的代理物件中，從而讓您的業務程式碼可以保持極致的純淨。

總結

在本篇文章中，我們為後續的學習奠定了堅實的理論基礎。

  • 我們深入複習了 ACID 原則，明確了可靠交易系統的四大目標。
  • 我們揭開了 @Transactional 背後的運作原理——AOP 動態代理，理解了 Spring 是如何透過「攔截」方法呼叫，來自動化地管理交易的生命週期。

理解了 AOP 代理這個核心原理，是解鎖下一個謎題的關鍵。既然是代理在外部進行攔截，那麼如果一個 @Transactional 方法是在類別內部被另一個方法所呼叫，它還會經過這個代理嗎？
