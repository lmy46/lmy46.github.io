Spring 核心 05 - 進階依賴注入與條件化組態

發表於2025-09-08|更新於2025-09-29|JavaSpringSpring 核心實戰
|字數總計:3k|閱讀時長:10分鐘|閱讀量:

前言：當依賴關係變得複雜

我們已經知道，Spring IoC 容器這位「總經理」，擅長為它的「員工」(Beans) 配發所需的「工具」(依賴)。但在一個龐大的公司裡，組織結構遠比想像的複雜：

  • 職位衝突：公司裡可能有多位「溝通專員」，一位擅長寄送 Email，另一位擅長發送簡訊。當一個部門需要「一位溝通專員」時，總經理該派誰去？
  • 循環指派：A 部門的運作需要 B 部門的報告，但 B 部門產出報告又需要 A 部門的數據，形成了死結。
  • 條件式成立：只有當公司在海外設有分部時，才需要成立「國際事務部」。

這些場景，對應的正是 Spring 依賴注入中的注入衝突、循環依賴和條件化組態等棘手問題。本篇文章將教您如何像一位資深架構師一樣，使用 Spring 提供的進階工具，優雅地解決這些問題。

解決注入衝突：@Primary 與 @Qualifier

問題：當一個介面有多個實作類別時，Spring 在進行依賴注入時會不知所措，因為它不知道該選擇哪一個實作類別的 Bean。

場景：假設我們有一個 NotificationSender 介面，以及兩個實作：

解決方案一：@Primary - 指定首選方案

@Primary 註解告訴 Spring：「如果在注入時發現有多個同類型的候選 Bean，請優先選擇我這一個」。

現在，OrderService 將會被順利注入 EmailSender 的實例。@Primary 非常適合用來設定一個最常用、最通用的預設實作。

解決方案二：@Qualifier - 精準指定名稱

@Qualifier 提供了比 @Primary 更精準的控制。它讓我們可以透過名稱，來明確指定要注入哪一個 Bean。

  1. 為 Bean 命名：@Component 預設的 Bean 名稱是類別名的小駝峰寫法（emailSender, smsSender）。
  2. 在注入點指定：

**@Qualifier 的優先級高於 @Primary**。即使 EmailSender 被標註為 @Primary，OrderService 也依然會被注入 SmsSender。

延遲戴入：@Lazy 的用途

1. 主要用途（作為解決方案）：打破循環依賴

問題：ServiceA 依賴 ServiceB，同時 ServiceB 也依賴 ServiceA。

當使用建構子注入時，這會導致一個無法解開的死結：為了建立 A，需要 B；為了建立 B，又需要 A。應用程式將啟動失敗。

解決方案：@Lazy 註解。
我們可以將 @Lazy 註解加在其中一個建構子的參數上，來打破這個循環。

運作原理：@Lazy 告訴 Spring：「在建立 ServiceA 的時候，先不要急著把 ServiceB 的完整實例給我。請先給我一個 ServiceB 的代理物件，讓我能順利完成建構。等到我之後真的需要呼叫 ServiceB 的方法時，你再透過這個代理物件，去找到那個真正的 ServiceB 實例。」

設計警訊：雖然 @Lazy 可以解決問題，但循環依賴通常是「程式碼設計不良」的一個警訊。最佳的解決方案，往往是重新審視您的類別職責，透過重構來解開這個循環。

重構程式碼：將兩個服務中共同依賴的邏輯，抽離到第三個服務 ServiceC 中。然後讓 ServiceA 和 ServiceB 都去依賴 ServiceC，從而將 A <=> B 的循環關係，變成 A => C 和 B => C 的單向關係。

只有在重構的成本非常高，或是在處理一些無法修改的舊系統時，@Lazy 才應該被當作解決循環依賴的最終手段。

2. 更佳用途（作為優化手段）：延遲載入重量級資源

這是在沒有循環依賴的情況下，一個更推薦、更「優雅」的用途：優化應用程式的啟動時間。

場景：
假設您有一個 HeavyResourceService，它在初始化的時候需要執行非常耗時的操作，例如：

  • 載入一個巨大的機器學習模型到記憶體中。
  • 與一個遠端的、反應很慢的系統建立連線。
  • 從磁碟讀取一個超大的設定檔並進行解析。

但是，這個服務在應用程式運行期間，可能很少被用到，也許只有在觸發某個特定管理功能時才會被呼叫。

問題：
在預設情況下，Spring 會在啟動時就建立所有 Singleton Bean。這會導致您的應用程式因為等待 HeavyResourceService 初始化完成，而啟動得非常緩慢。

@Lazy 的解決方案：

效果：

  • 應用程式啟動時，Spring 不會去建立 HeavyResourceService 的真實物件，所以啟動速度會非常快。
  • 當使用者第一次訪問 /do-heavy-work 這個端點時，代理物件會攔截到 doWork() 的呼叫，此時它才會去建立真實的 HeavyResourceService 物件（這時會等待 5 秒鐘），然後再執行 doWork() 方法。
  • 之後再次訪問，因為真實物件已經存在，所以會直接執行，不再有延遲。

總結 @Lazy 的真正用途：
它的首要目標是延遲初始化。這個特性可以用來打破循環依賴（被動用途），也可以用來優化啟動性能（主動用途），而後者通常是更健康的應用場景。

條件化組態：@Conditional 的魔力

這是 Spring Boot 自動設定功能背後的終極武器。@Conditional 系列註解允許一個 Bean 或一個組態，只有在滿足特定條件時，才會被建立與註冊。

常用條件註解：

  • @ConditionalOnProperty(name = "...", havingValue = "...")：只有當 application.yml 中的某個屬性存在且值匹配時，才會生效。
  • @ConditionalOnClass(...)：只有當 classpath 上存在某個特定的類別時，才會生效。
  • @ConditionalOnBean(...)：只有當 IoC 容器中已存在某個指定的 Bean 時，才會生效。
  • @ConditionalOnMissingBean(...)：只有當 IoC 容器中不存在某個指定的 Bean 時，才會生效（常用於提供預設實作）。

實戰範例：一個可透過設定檔開關的功能

1. @ConditionalOnProperty：根據設定檔屬性決定

  • 用途：當 application.yml 或 application.properties 中的某個屬性值符合條件時，才建立這個 Bean。最常用於功能開關。
  • 場景：我們希望提供一個訊息推播功能，但允許在設定檔中決定是使用正式的推播服務，還是僅在日誌中印出模擬訊息。

2. @ConditionalOnClass：根據 Classpath 是否存在某個類別決定

  • 用途：當專案的依賴中包含了某個特定的類別時，才建立這個 Bean。這是 Spring Boot 自動設定的核心，用來整合第三方函式庫。
  • 場景：我們正在開發一個資料庫監控模組。我們希望只有當使用者加入了 MySQL 的 JDBC 驅動程式依賴時，才自動設定一個專門用於 MySQL 的健康檢查器。

3. @ConditionalOnBean：根據容器中是否已存在某個 Bean 決定

  • 用途：當 IoC 容器中已經存在了另一個指定的 Bean 時，才建立這個 Bean。常用於基於已有的 Bean 進行擴充設定。
  • 場景：我們有一個通用的 DataSource (資料來源) Bean。我們希望只有當這個 DataSource Bean 成功被建立後，才為它設定一個資料庫連線池的監控儀表板 ConnectionPoolDashboard。

4. @ConditionalOnMissingBean：根據容器中是否不存在某個 Bean 決定

  • 用途：當 IoC 容器中沒有指定類型的 Bean 時，才建立這個 Bean。最常用於提供一個可被使用者覆寫的預設實作。
  • 場景：我們提供一個預設的錯誤處理器 DefaultErrorHandler。但我們允許應用程式的開發者自己定義一個更客製化的 CustomErrorHandler。如果開發者定義了，就用他的；如果他沒定義，就用我們預設的。

透過這種方式，我們可以建立出高度靈活、可插拔的模組。整個 Spring Boot 的自動設定，就是基於這套強大的條件化組態機制所建構的。

旅程總結：從使用者到掌控者

恭喜您完成了「Spring 核心實戰」系列的全部篇章！

在這五篇文章的旅程中，我們一起深入了 Spring 框架的心臟地帶。我們從 Bean 的定義開始，理解了元件掃描與 Java 組態的區別；接著，我們探索了 Bean 的不同生命型態 (Scope) 與生命週期 (@PostConstruct)；最終，我們掌握了解決依賴注入衝突 (@Qualifier)、循環依賴 (@Lazy) 以及實現條件化組態 (@Conditional) 的進階技巧。
